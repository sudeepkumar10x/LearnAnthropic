SECTION 1: EXAM STRUCTURE & FORMAT
================================================================================

QUESTION DISTRIBUTION:
- ~5 questions follow the same format/structure as the Anthropic practice exam
- Rest of the questions (majority) are different in both format and content
- Do NOT rely solely on memorizing practice exam questions
- Focus on understanding concepts deeply, not pattern-matching

SCENARIO CHANGES:
- Scenario names/titles may differ from what you've studied
- The underlying architecture patterns and concepts remain consistent
- Example: A "Customer Support Resolution Agent" might appear as "Support Case Orchestrator"
- Study the pattern, not just the name

KEY TAKEAWAY: Prepare by mastering core concepts and patterns, not by memorizing 
specific questions or scenario names.

================================================================================
SECTION 2: CRITICAL EXAM TOPICS & PATTERNS
================================================================================

MULTI-AGENT ORCHESTRATION:
- Coordinator agent patterns are heavily tested
- Common failure scenario: Coordinator calls subagent but gets no result back
  * Watch for cases where the tool call is visible/logged but execution fails
  * Ask yourself: Is the subagent tool properly configured?
  * Check: allowedTools, tool description routing, tool availability in subagent context
  * Debug: Distinguish between "tool called but no result" vs "tool never called"

CONVERSATION API:
- Understand when to use Conversation API vs Batch API
- API choice impacts agent responsiveness and error handling
- Know the trade-offs for real-time vs fire-and-forget patterns

PROMPT ENGINEERING & STRUCTURED OUTPUT:
- This domain appears heavily on the exam
- Be strong with JSON schema extraction and validation
- Know when to enforce structured output vs allow flexibility

HOOKS & FINANCIAL TRANSACTIONS:
- PostToolUse hooks are frequently tested
- Examples involving money/refunds/payments are important
  * Refund processing workflows
  * Payment validation and error handling
  * Financial transaction auditing
- Understand why hooks matter in financial contexts:
  * Normalization of tool outputs before passing to next agent
  * Validation before state changes
  * Audit trail and compliance
- Short examples with real financial stakes are often in questions

SECTION 4: TECHNICAL DEEP DIVES TO MASTER
================================================================================

COORDINATOR & SUBAGENT PATTERNS:
- Explicit context passing: Know what information flows to subagents
- Isolation: Understand when subagents need isolated tool sets (allowedTools)
- Failure modes: Coordinator calls tool → tool shows as called → no result
  * This happens when:
    - Tool configuration error in subagent
    - Tool not in allowedTools list for that subagent
    - Tool execution timeout without proper error handling
    - Missing error propagation back to coordinator

STOP_REASON & CONTROL FLOW:
- Master stop_reason interpretation (tool_use, end_turn, max_tokens, stop_sequence)
- Understand how stop_reason signals orchestration decisions
- Know when to retry, escalate, or terminate

POSTTOOLUSE HOOKS:
- Learn when/why to normalize tool outputs with hooks
- Financial transactions are a key use case:
  * Validate refund amounts before processing
  * Normalize payment response formats
  * Audit trail creation
- Hooks prevent downstream agents from working with malformed data

CONTEXT MANAGEMENT:
- Lost-in-the-middle problem: How to structure long context windows
- Token efficiency while maintaining accuracy
- Case facts blocks to prevent lossy summarization

SECTION 5: EXAM DAY LOGISTICS & BEST PRACTICES
================================================================================

TECHNICAL SETUP (Critical - Do This in Advance):
✓ Plug in your laptop/charger - Do NOT rely on battery during exam
✓ Test your internet connection - Stable, hardwired connection preferred
✓ Set up proctor software in advance:
  - Download and test any required proctoring tools before exam day
  - Verify camera, microphone, and screen sharing work
  - Know how to share your screen if asked
  - Have proctor-required information ready (ID, workspace setup)

ENVIRONMENT & COMMUNICATION:
✓ Find a quiet location - No background noise, no interruptions
✓ Clear camera view - Proctor needs clear view of you and workspace
✓ Clean background - Professional appearance, no clutter
✓ Stable camera angle - Avoid awkward positioning
✓ Mute notifications - Phone, Slack, email, calendar alerts
✓ Close unnecessary windows - Only exam window + notepad if allowed
✓ Have water available - Long exam, stay hydrated

EXAM DAY MINDSET:
- Read questions carefully - Scenario names may differ, but patterns are familiar
- Don't memorize - Understand and apply concepts
- Flag and return - If stuck, move on and come back
- Check your work - Review answers before submitting
- Manage time - Don't spend 10+ minutes on one question

SECTION 7: COMMON PITFALLS TO AVOID
================================================================================

1. Assuming all questions match practice exam format
   → Real exam has 5 similar + many different questions
   → Focus on understanding, not memorization

2. Missing scenario name changes
   → Scenario might be called something different
   → Study the pattern and concepts, not the name

3. Not debugging coordinator-subagent failures
   → "Tool called but no result" is a specific failure mode
   → Check: tool config, allowedTools, error handling
   → Ask: Is the subagent even seeing the tool?

4. Overlooking financial transaction examples
   → Refunds, payments, transactions appear frequently
   → Understand why they need hooks and validation
   → Study real-world financial orchestration patterns

5. Poor exam environment setup
   → Proctor issues slow you down or disqualify you
   → Set up in advance, test everything
   → Clear environment, stable camera, quiet space

6. Not managing time effectively
   → Some questions are longer/complex
   → Don't get stuck on one question
   → Flag, move on, return if time allows

SECTION 10: KEY INSIGHTS SUMMARY
================================================================================

TOP 5 EXAM INSIGHTS:

1. SCENARIO NAMES CHANGE
   Not all scenarios are named the same in the exam. Study patterns and 
   architectural concepts, not scenario names.

2. COORDINATOR-SUBAGENT FAILURES ARE TESTED
   "Tool called but no result" is a specific failure mode. Understand when 
   this happens and how to debug it.

3. FINANCIAL TRANSACTIONS MATTER
   Refunds, payments, and money-involved examples appear frequently. 
   Understand why hooks are critical here.

4. POSTTOOLUSE HOOKS ARE IMPORTANT
   Hooks normalize outputs and validate state changes. They're especially 
   critical in orchestration and financial workflows.

5. EXAM ENVIRONMENT SETUP IS CRITICAL
   Charge your laptop, test proctor software in advance, have a quiet 
   environment with clear camera view. These logistics prevent last-minute 
   stress.
